## 一、本周概述

本周工作围绕 F1TENTH 两个月落地方案中的 W3 阶段展开，重点完成创新点 1：基于 MID-360 三维雷达的多层感知节点。此前 W1 管线只使用 `pointcloud_to_laserscan` 从三维点云中截取一层薄高度切片并压成 2D `/scan`，高度信息被丢弃；本周新增 `multilayer_scan` 包，将 `/livox/lidar` 点云按高度带投影成多条 `LaserScan`，再融合为 `/scan_3d` 供反应式规划器使用。

核心结论：W3 软件实现与离线逻辑验证已完成，已交付多层感知节点、配置文件、单独启动 launch、W1+W2+W3 全栈 launch 和使用指南；后续还需要在实车上完成 `colcon build`、RViz 分层验证和 `/scan_3d` 低速闭环跑通。

## 二、本周完成

### 1. 明确 W3 设计原则

本周首先确认当前管线没有地面分割，现有单层切片 `[-0.05, 0.10] m` 能稳定工作的原因，是它贴近雷达高度并避开大部分地面回波。因此 W3 多层感知不能破坏 W1 已经跑通的单层行为。

| 设计点 | 结论 | 作用 |
| ---- | ---- | ---- |
| 保留现有工作切片 | `body` 高度带设为 `[-0.05, 0.10] m` | 保证 `/scan_3d` 至少包含 W1 单层 `/scan` 的阻挡信息 |
| 多层高度划分 | 默认分为 `low`、`body`、`high` 三层 | 区分低矮障碍、车体高度障碍和悬空物 |
| 融合策略 | 各阻挡层逐角度取最近距离 | 让规划器更谨慎，不比 W1 更危险 |
| 悬空层处理 | `high` 默认 `blocking=false` | 避免门楣、桌沿等高处回波变成假墙 |

> 关键判断：`/scan_3d` 的阻挡层融合结果应当不回归 W1；它只增加必要的保守性，不应漏掉原本单层能看到的车体高度障碍。

### 3. 完成点云解析与投影逻辑

本周确认 Livox MID-360 的输入为标准 `sensor_msgs/PointCloud2`，`/livox/lidar` 使用 `livox_frame`，发布频率配置为 `10 Hz`。由于当前环境没有 `sensor_msgs_py`，节点采用 `numpy` 结构化 dtype 从 `PointCloud2` 原始 buffer 中按字段名解析 `x/y/z`，对 Livox PointXYZRTL 的额外字段保持鲁棒。

多层投影逻辑如下：

| 步骤 | 处理内容 |
| ---- | ---- |
| 解析点云 | 从 `PointCloud2` 的字段中提取 `x/y/z` |
| 自车体过滤 | 丢弃 `|x| < 0.30` 且 `|y| < 0.20` 的近车体点 |
| 极坐标投影 | 计算 `range = hypot(x, y)` 和 `angle = atan2(y, x)` |
| 高度分层 | 按 `z` 落入 `low/body/high` 高度带 |
| 每层取最近 | 每层每个角度 bin 保留最近距离 |
| 阻挡融合 | 对 `blocking=true` 的层逐角度取最小距离 |

默认高度带：

| 层 | 高度范围 | 是否阻挡 | 用途 |
| ---- | ---- | ---- | ---- |
| `low` | `[-0.18, -0.05) m` | 是 | 低矮障碍、路肩、低箱体 |
| `body` | `[-0.05, 0.10) m` | 是 | 车体高度障碍，复现 W1 单层切片 |
| `high` | `[0.10, 0.50) m` | 否 | 门楣、桌沿等悬空物调试层 |

### 4. 修复角度边界问题并完成离线验证

离线合成点云验证时发现，`angle=π` 附近的边界点可能因为 360 度边界 bin 处理导致缺失。该问题已通过对角度索引执行 modulo 回绕修复。

| 验证项 | 结果 |
| ---- | ---- |
| Python 语法检查 | `multilayer_scan_node.py`、两个 launch 文件和 `setup.py` 均通过 `py_compile` |
| 合成 Livox 风格 buffer 解析 | 能正确解析 `x/y/z` |
| 自车体过滤 | 近车体点被过滤 |
| 高度分层 | `low/body/high` 各命中 1 个测试点 |
| 悬空层排除 | `high` 层命中但不进入融合阻挡层 |
| 融合结果 | 有效点 3 个，融合阻挡角点 2 个，符合预期 |
| 角度回绕 | 修复 `±π` 边界点丢失问题 |

> 当前验证边界：已完成代码语法与合成数据逻辑验证；真实 MID-360 点云、RViz 分层显示和实车低速闭环仍需上车验证。

## 四、问题与风险

| 问题 | 现象 | 根因 | 影响 | 当前应对 |
| ---- | ---- | ---- | ---- | ---- |
| 尚无地面分割 | 地面可能进入 `low/body` 造成假墙 | 当前 W3 仅用高度带避开地面，不估计真实地面平面 | 复杂坡面或不平地可能影响 `/scan_3d` | RViz 中调 `band_floor/band_ceil`，后续 W5 可加入轻量地面估计 |
| 高度带需要实车调参 | 默认参数未经过现场 RViz 校准 | MID-360 安装高度、姿态和场地地面会影响 z 分布 | 可能漏检低矮障碍或误报地面 | 使用 W3 指南中的 RViz 对照步骤逐层调参 |
| `/scan_3d` 尚未实车闭环验证 | 软件链路已写好，但规划器实际反应未验证 | 离线合成点云不能替代真实点云时序和噪声 | 暂不能声称 W3 实车完成 | 先单独跑多层节点，再接 `race_3d.launch.py` 低速测试 |
| 悬空层是否阻挡需按场景决策 | `high` 默认不阻挡 | 为避免悬空物假墙，但某些高障碍可能仍会碰车 | 场景变化时策略可能过松或过紧 | 保持 `high=false` 首跑，必要时新增中高层或调整 blocking |

## 六、本周结论

本周完成了 W3 的核心软件交付：`multilayer_scan` 已能把 MID-360 三维点云按高度带投影为多层 scan，并融合为 `/scan_3d` 接入反应式规划器链路。设计上保留 `body=[-0.05,0.10]` 作为 W1 单层切片，融合时只对阻挡层取最近距离，保证多层感知不会主动削弱已有单层能力。

当前 W3 处于“软件完成、实车验证待执行”状态。下一步重点不是继续堆功能，而是在 RViz 中完成高度带标定，并用 `race_3d.launch.py` 进行低速闭环验证，确认 `/scan_3d` 的感知收益真实存在且不引入地面假墙。
