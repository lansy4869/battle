# W2 使用指南：系统辨识补全 + g-g-v 安全壳

> 适用范围：`~/下载/studies/slash_ws/jetson_ws`，车上通常为 `~/jetson_ws`。  
> 目标：完成 W2 的“录包 -> 系统辨识 -> 回填安全壳 -> 低速验证”闭环。  
> 交付包：
> - `slash_sysid`：底盘系统辨识与绕圆横向极限标定。
> - `slash_safety`：g-g-v 物理安全护盾，将 `/drive_raw` 投影到安全的 `/drive`。

---

## 0. W2 要解决什么

W1 已经完成实车低速反应式跑通。本周 W2 的目标是给上游规划器和底盘之间加一层可解释的物理安全壳：

```
battle_fast2 --(/drive_raw)--> ggv_shield --(/drive)--> ackermann_mux(navigation) --> vesc
                              ^
                              |
                            /odom
```

安全壳的约束来自系统辨识：

| 量 | 来源 | 当前值/状态 | 用途 |
| ---- | ---- | ---- | ---- |
| `ax_accel_max` | 纵向 bag 实测 | `6.35 m/s^2` | 限制加速斜率 |
| `ax_brake_max` | 纵向 bag 实测 | `6.66 m/s^2`，取正幅值 | 限制制动斜率与失效刹停 |
| `steering_limit_rad` | servo 行程标定 | `0.34 rad` | 转向限幅 |
| `wheelbase` | 当前配置/绕圆估计 | 初始 `0.25 m` | 曲率计算 |
| `ay_max` | 绕圆 bag 实测 | 录包前暂用 `9.81 m/s^2` 假设 | 过弯限速 |

关键公式：

```text
kappa = tan(|delta|) / L
v_cap = min(v_max, sqrt(ay_max * ay_safety_factor / kappa))
```

其中 `ay_safety_factor` 首次实车验证建议保持 `0.6`，稳定后再逐步放开。

## 2. 前置检查

### 2.1 确认底盘 bringup 已运行

先启动 W1 已经跑通的整车 bringup，确保 VESC、里程计、mux、手柄接管都在。

需要看到这些关键话题：

```bash
ros2 topic list | grep -E '^/(drive|teleop|odom|sensors/imu|commands/servo/position|commands/motor/speed|ackermann_cmd)$'
```

至少应具备：

| 话题 | 作用 |
| ---- | ---- |
| `/drive` | mux 的 navigation 输入；绕圆采集时 `circle_commander` 默认发这里 |
| `/teleop` | 手柄输入，mux 中优先级 100，高于 navigation |
| `/odom` | VESC 轮式里程计，系统辨识和安全壳都要用 |
| `/sensors/imu` | VESC IMU，绕圆时用于实测横向加速度 |
| `/commands/servo/position` | 实际 servo duty，用于转向角换算 |
| `/commands/motor/speed` | 电机速度命令，用于纵向辨识保留 |
| `/ackermann_cmd` | mux 输出到底盘链路，可作为录包参考 |

### 2.2 确认 mux 优先级

当前 `f1tenth_stack/config/mux.yaml` 中：

```yaml
navigation:
  topic: drive
  priority: 10
joystick:
  topic: teleop
  priority: 100
```

因此绕圆录包时，`circle_commander` 发 `/drive`，手柄 `/teleop` 可以随时抢占。实车采集必须保证手柄可用，人在车旁可以立即接管。

### 2.3 确认场地与参数

首次绕圆建议：

| 参数 | 建议值 | 说明 |
| ---- | ---- | ---- |
| `steering_angle` | `0.20` 到 `0.25 rad` | 先保守，确认半径足够再用 `0.25` |
| `v_target` | `1.0` 到 `1.2 m/s` | 首次不要直接上高速 |
| `hold_time` | `15 s` | 保持足够样本 |
| `direction` | `1.0` 和 `-1.0` | 建议左右各录一段 |

## 3. 录绕圆 bag

### 3.1 终端 1：启动绕圆指令

进入工作区并 source：

```bash
cd ~/jetson_ws
source install/setup.bash
```

先低速录第一段：

```bash
ros2 launch slash_sysid circle_test.launch.py \
  steering_angle:=0.25 \
  v_target:=1.2 \
  ramp_time:=3.0 \
  hold_time:=15.0 \
  direction:=1.0
```

参数含义：

| 参数 | 含义 |
| ---- | ---- |
| `drive_topic` | 默认 `/drive`，直接走 mux navigation 口 |
| `steering_angle` | 固定转向角，单位 rad |
| `v_target` | 目标速度，单位 m/s |
| `ramp_time` | 从 0 缓升到目标速度的时间 |
| `hold_time` | 保持目标速度的时间 |
| `direction` | `1.0` 为一侧转向，`-1.0` 为反向 |

### 3.2 终端 2：同步录包

在启动绕圆前或同时开始录包：

```bash
cd ~/jetson_ws
ros2 bag record \
  /odom \
  /sensors/imu \
  /commands/servo/position \
  /commands/motor/speed \
  /ackermann_cmd \
  -o circle_test_left
```

第一段结束后停止录包。建议再录反向一段：

```bash
ros2 launch slash_sysid circle_test.launch.py \
  steering_angle:=0.25 \
  v_target:=1.2 \
  ramp_time:=3.0 \
  hold_time:=15.0 \
  direction:=-1.0
```

对应录包：

```bash
cd ~/jetson_ws
ros2 bag record \
  /odom \
  /sensors/imu \
  /commands/servo/position \
  /commands/motor/speed \
  /ackermann_cmd \
  -o circle_test_right
```

### 3.3 实车安全要求

- 场地必须空旷，先确认转向半径不会碰到边界。
- 手柄保持在线，随时通过 `/teleop` 抢占。
- 发现车身不稳、打滑、逼近边界或定位异常，立即手柄接管。
- 首次验证不追求极限，只要求得到车速大于 `0.5 m/s`、转角持续、IMU 有数据的有效 bag。

## 4. 检查 bag 是否有效

### 4.1 查看录包话题

```bash
cd ~/jetson_ws
ros2 bag info circle_test_left
```

必须包含：

```text
/odom
/sensors/imu
/commands/servo/position
```

建议同时包含：

```text
/commands/motor/speed
/ackermann_cmd
```

### 4.2 用辨识脚本做快速检查

```bash
cd ~/jetson_ws
python3 src/slash_sysid/slash_sysid/chassis_sysid.py \
  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out/circle_left_check
```

如果输出中出现以下字段，说明横向数据链路可用：

```json
"lateral": {
  "n_corner_samples": ...,
  "v_corner_max_mps": ...,
  "imu_accel_units_used": "...",
  "identifiable": true,
  "ay_lat_max_mps2_MEASURED": ...
}
```

判定标准：

| 字段 | 合格条件 |
| ---- | ---- |
| `n_corner_samples` | 大于 `10`，越多越稳 |
| `v_corner_max_mps` | 大于 `0.5` |
| `identifiable` | 必须为 `true` |
| `ay_lat_max_mps2_MEASURED` | 存在且为正数 |
| `imu_accel_units_used` | `auto` 判定后显示 `g` 或 `mps2` |

如果 `identifiable=false`，优先检查：车速是否太低、转角是否持续、是否录到了 `/sensors/imu`。

### 5.1 单段绕圆辨识

```bash
cd ~/jetson_ws

  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out
```

输出文件：

| 文件 | 说明 |
| ---- | ---- |
| `src/slash_sysid/out/identified_params.json` | 系统辨识参数 |
| `src/slash_sysid/out/sysid_overview.png` | 纵向拟合与 g-g-v 概览图 |
| `src/slash_sysid/out/speed_profile.png` | 曲率到限速的 profile |

### 5.2 IMU 单位强制指定

脚本默认 `--imu_accel_units auto`。如果发现 `ay_max` 明显过大或过小，可以手动指定：

```bash

  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out \
  --imu_accel_units g
```

或：

```bash
python3 src/slash_sysid/slash_sysid/chassis_sysid.py \
  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out \
  --imu_accel_units mps2
```

### 5.3 结果字段解释

重点看 `identified_params.json` 中这几段：

```json
"longitudinal": {
  "K": 0.9138,
  "tau_s": 0.255,
  "Td_s": 0.0,
  "fit_r2": 0.973,
  "ax_accel_max_mps2": 6.35,
  "ax_brake_max_mps2": -6.66
}
```

纵向部分已经由历史 bag 实测，主要用于给安全壳提供纵向变化率限制。

```json
"lateral": {
  "identifiable": true,
  "ay_lat_max_mps2_MEASURED": 7.8,
  "wheelbase_eff_m_MEASURED": 0.26,
  "mu_eff_MEASURED": 0.79
}
```

横向部分来自本次绕圆 bag。实际数值以你的输出为准。

```json
"g_g_v": {
  "ax_long_max_mps2_MEASURED": 6.66,
  "ay_lat_max_mps2": 7.8,
  "ay_source": "MEASURED (circle bag IMU)",
  "vmax_mps": 2.0
}
```

只有当 `ay_source` 为 `MEASURED (circle bag IMU)` 时，才表示横向极限已经从假设值切换为实测值。

## 6. 把辨识结果灌进护盾

目标文件：

```bash
src/slash_safety/config/ggv_shield.yaml
```

需要回填的字段：

| YAML 字段 | 从 JSON 哪里取 | 备注 |
| ---- | ---- | ---- |
| `ay_max` | `g_g_v.ay_lat_max_mps2` | 必须优先使用实测横向极限 |
| `ax_accel_max` | `longitudinal.ax_accel_max_mps2` | 正数 |
| `ax_brake_max` | `abs(longitudinal.ax_brake_max_mps2)` 或 `g_g_v.ax_long_max_mps2_MEASURED` | YAML 中填正幅值 |
| `wheelbase` | `lateral.wheelbase_eff_m_MEASURED` | 若不存在或异常，保留 `0.25` |
| `steering_limit_rad` | `steering.steering_angle_span_rad` 的最大绝对值 | 当前默认 `0.34` |
| `v_max` | `g_g_v.vmax_mps` | 首跑保持 `2.0` |
| `ay_safety_factor` | 人工设置 | 首跑建议 `0.6` |

示例：

```yaml
ggv_shield_node:
  ros__parameters:
    wheelbase: 0.25
    ay_max: 7.80
    ay_safety_factor: 0.6
    ax_accel_max: 6.35
    ax_brake_max: 6.66
    steering_limit_rad: 0.34
    v_max: 2.0
```

注意：

- `ax_brake_max` 在 JSON 的 `longitudinal.ax_brake_max_mps2` 是负数，写进 YAML 时必须取正值。
- `ay_max` 如果仍来自 `ASSUMED (mu=1.0)`，论文和总结中要标注“横向仍为假设值”。
- `ay_safety_factor` 不要因为测到了 `ay_max` 就直接改成 `1.0`，实车首跑仍保持 `0.6`。

## 8. 护盾功能验证

### 8.1 不接规划器的静态验证

先只启动护盾：

```bash
ros2 launch slash_safety ggv_shield.launch.py
```

另开终端发布一个故意过激的原始指令：

```bash
ros2 topic pub -r 20 /drive_raw ackermann_msgs/msg/AckermannDriveStamped \
  "{drive: {steering_angle: 0.34, speed: 5.0}}"
```

观察输出：

```bash
ros2 topic echo /drive
ros2 topic echo /ggv_shield/v_cap
```

合格现象：

- `/drive` 中 `steering_angle` 不超过 `0.34`。
- `/drive` 中 `speed` 会被压到 `/ggv_shield/v_cap` 附近，不会直接输出 `5.0`。
- 停止发布 `/drive_raw` 后，`/drive` 会在 `input_timeout_s=0.3` 后进入平滑刹停并转向回中。

### 8.2 接入反应式规划器验证

前提：W1 的雷达、p2l、bringup、`/scan`、`/odom` 已经在跑。

启动“规划器 + 护盾”：

```bash
ros2 launch slash_safety race_with_shield.launch.py
```

该 launch 会让：

```text
battle_fast2 -> /drive_raw
ggv_shield  -> /drive
```

检查话题：

```bash
ros2 topic hz /drive_raw
ros2 topic hz /drive
ros2 topic echo /ggv_shield/v_cap
```

低速实车验证顺序：

1. 原地或架空先看 `/drive_raw` 和 `/drive` 是否都有输出。
2. 小车落地，`v_max` 保持 `2.0`，`ay_safety_factor` 保持 `0.6`。
3. 在空旷场地低速跑，观察大转角时 `/drive` 速度是否低于 `/drive_raw`。
4. 关闭规划器，确认安全壳能在上游超时后刹停。
5. 手柄 `/teleop` 接管测试，确认 mux 高优先级仍可抢占。

## 10. 常见问题排查

### 10.1 `identifiable=false`

常见原因：

- 车速没有超过 `0.5 m/s`。
- 转角持续时间太短。
- `/sensors/imu` 没录到或无法反序列化。
- 场地太小，车还没进入稳定绕圆就停止。

处理：

- 把 `hold_time` 提到 `20 s`。
- 确认 `ros2 bag info` 中有 `/sensors/imu`。
- 先保持 `v_target=1.2`，不要一开始就追求更高速度。

### 10.2 `ay_max` 明显不合理

常见原因：

- IMU 线加速度单位判断错误。
- 录包时车身有明显冲击、急停或手柄抢占。
- 绕圆数据中混入非稳态阶段太多。

处理：

```bash
python3 src/slash_sysid/slash_sysid/chassis_sysid.py \
  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out \
  --imu_accel_units g
```

如果仍异常，再用：

```bash
python3 src/slash_sysid/slash_sysid/chassis_sysid.py \
  --bags . \
  --circle_bag ./circle_test_left \
  --out src/slash_sysid/out \
  --imu_accel_units mps2
```

### 10.3 安全壳没有输出 `/drive`

检查：

```bash
ros2 topic hz /drive_raw
ros2 topic hz /odom
ros2 node info /ggv_shield_node
```

可能原因：

- 上游没有发 `/drive_raw`。
- launch 中 `drive_in_topic` 配错。
- 安全壳未 source 最新 install。

### 10.4 `/drive` 输出没有被限速

检查：

- `ggv_shield.yaml` 中 `ay_max` 是否过大。
- `ay_safety_factor` 是否被改成 `1.0` 或更高。
- 发布测试时 `steering_angle` 是否接近 0；直线情况下曲率很小，限速不会明显触发。

## 11. 写进论文或周报的诚实状态

| 项目 | 状态 | 可写结论 |
| ---- | ---- | ---- |
| 纵向模型 | 已实测 | `K=0.914`，`tau=0.255 s`，`Td=0 s`，`R^2=0.97` |
| 加减速极限 | 已实测 | 加速 `+6.35 m/s^2`，制动 `-6.66 m/s^2` |
| 转向行程 | 已标定 | servo 行程对应约 `+/-0.34 rad` |
| 横向极限 | 绕圆后实测 | 只有 `ay_source=MEASURED` 时才能写“实测” |
| 安全壳 | 软件完成，待实车闭环验证 | 可解释物理约束层，独立于上游策略 |

如果还没有完成绕圆录包，表述应为：

> g-g-v 安全壳已实现并接入控制链路，纵向极限来自实车 bag；横向极限在绕圆 bag 完成前暂采用 `mu*g` 假设，并在配置与文档中显式标注。

如果绕圆录包已经完成且 `ay_source=MEASURED`，表述可改为：

> g-g-v 安全壳已使用本车绕圆 bag 标定的横向极限 `ay_max`，实现了基于实测车辆可行域的转向限幅、过弯限速、纵向变化率限制与失效刹停。
